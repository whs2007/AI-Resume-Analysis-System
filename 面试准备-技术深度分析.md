### 1.2 错误处理的完整链路

**PDF 上传** (`pdf_parser.py:22-35`)：
```
用户上传 → 后缀检查(.pdf) → 空文件检查 → 大小检查(10MB) 
→ PDF 魔数检查(%PDF) → PyMuPDF 打开 → 逐页提取 → 文本清洗
→ 空文本检测（图片PDF拒绝）
```
每一步失败都有**中文错误提示**，分别对应不同状态码。

**AI 降级链路** (`ai_extractor.py:92-114`)：
```
AI 调用 (qwen-turbo) 
  → 成功: 解析 JSON → 返回结构化数据
  → JSON 解析失败: 重试(最多2次)
  → 全部失败: 降级为正则规则提取(邮箱、电话、姓名)
```

**缓存降级链路** (`cache.py:67-78`)：
```
Redis 可用 → Redis 读写
Redis 不可用 → InMemoryStore(dict + TTL)
```
`available` 属性永远返回 `True`，调用方无需判断。


## 二、代码质量

### 2.1 分层架构

```
routers/          ← 薄层：参数校验、HTTP 响应，不写业务逻辑
    ↓
services/         ← 厚层：PDF 解析、AI 提取、匹配、缓存
    ↓
models/           ← Pydantic：请求/响应 schema，自动校验类型
    ↓
utils/            ← 纯函数：文本清洗、正则提取
```

**为什么这样分？** router 不碰 AI 调用逻辑，service 不碰 HTTP 细节。换 AI 模型只需改 `ai_extractor.py` 的 `model` 参数，API 层不受影响。

### 2.2 关键设计模式

**1. 门面模式 — CacheService (`cache.py`)**

对外暴露统一接口 `get_resume` / `set_resume` / `get_match` / `set_match`，内部自动选择 Redis 或内存，调用方无需关心底层实现。

```python
# cache.py:22-28 — InMemoryStore 数据结构
class InMemoryStore:
    def __init__(self):
        self._store: dict[str, tuple[float, dict]] = {}  # key → (过期时间戳, 数据)
    
    def get(self, key: str) -> Optional[dict]:
        entry = self._store.get(key)
        if entry is None: return None
        expires_at, data = entry
        if time.time() > expires_at:   # 惰性过期清理
            del self._store[key]
            return None
        return data
```

**2. 策略模式 — 匹配评分 (`matcher.py`)**

AI 评分和规则评分是两个可互换的策略：
- `match_resume_with_job()` — AI 策略（主）
- `_rule_based_match()` — 规则策略（降级）

```python
# matcher.py:147-149 — 综合评分公式
score = (
    details.skill_match * 0.5          # 技能权重最高
    + details.experience_relevance * 0.3  # 经验次之
    + details.education_match * 0.2       # 学历参考
)
```

**3. 模板方法 — AI 调用**

`ai_extractor.py` 和 `matcher.py` 都遵循同一模式：
```python

# 2. 构建 user prompt（注入数据）
# 3. Generation.call(temperature=0.1)  # 低温度保稳定
# 4. 解析 JSON（Markdown 剥离 + 首尾括号匹配）
# 5. except → 降级
```

### 2.3 Python import 的坑（面试加分点）

这是部署过程中遇到的一个经典 bug：

```python
# ❌ 错误写法 —— from-import 是值绑定
from app.services.cache import cache_service  # 永远 = None

# ✅ 正确写法 —— 模块引用是动态查找
import app.services.cache as cache_mod
svc = cache_mod.cache_service  # 运行时取最新值
```

**原因**：`from X import Y` 在 import 时刻将 `Y` 对象绑定到本地名。之后 `X.Y = new_value` 不会更新本地名。

**哪里会触发**：FastAPI 的 `lifespan` 是异步的，router 模块在 lifespan 之前就 import 了。如果 router 用 `from-import`，拿到的永远是 lifespan 执行前的 `None`。

这个 bug 在 commit `8a5dc80` 中修复，可以作为 "工程中踩坑并解决" 的例子。

## 四、技术深度

### 4.1 AI 模型选型：为什么通义千问

| 考虑因素 | 通义千问 | OpenAI GPT | 
|---------|---------|------------|
| 中文简历解析 | ⭐⭐⭐⭐⭐ 原生支持 | ⭐⭐⭐⭐ 需要英文 prompt |
| 网络延迟 | 阿里云 FC 内网，<50ms | 需要代理/跨境，>200ms |
| 成本 | 免费额度 + 按量付费 | 仅按量，无免费额度 |
| 模型替换 | DashScope SDK 统一接口 | —— |

**如何切换模型**：只需改 `ai_extractor.py:72` 和 `matcher.py:102` 的 `model="qwen-turbo"` 为 `"qwen-plus"` 或 `"qwen-max"` 即可获得更强的推理能力。

### 4.2 Prompt 工程深度分析

**信息提取 Prompt 设计 (`ai_extractor.py:16-38`)**：

```
你是专业的简历解析专家          ← 角色设定：缩小 AI 行为空间
按以下 JSON 格式返回            ← 明确输出格式
不要添加任何其他内容             ← 防止 AI 啰嗦
字段规则逐一说明                ← 边界情况处理
temperature=0.1                 ← 降低随机性，保证输出稳定
```

**JSON 解析的容错 (`ai_extractor.py:48-64`)**：

```python
def _parse_ai_response(text: str) -> dict:
    # 1. 剥离 Markdown 代码块 ```json ... ```
    code_match = re.search(r'```(?:json)?\s*\n?(.*?)\n?```', text, re.DOTALL)
    
    # 2. 找到第一个 { 和最后一个 }，截取 JSON 片段
    start = text.find('{')
    end = text.rfind('}')
    text = text[start:end + 1]
    
    # 3. 用 json.loads 验证合法性
    return json.loads(text)
```

这套解析兼容 AI 可能输出的三种格式：裸 JSON、Markdown 包裹、文本中嵌入。

### 4.3 缓存策略详解

```
缓存 Key 设计:
  简历缓存:  resume:{PDF文件MD5}          → 相同文件只解析一次
  匹配缓存:  match:{PDF_MD5}:{JD_MD5}     → 相同简历+相同JD只评分一次

过期策略:
  TTL = 24小时                           → 过期自动清理
  
降级策略:
  Redis 连接成功 → Redis 存储 + 内存备份
  Redis 连续失败 → 纯内存存储 (InMemoryStore)
```

**为什么两级缓存（Redis + 内存）？**
- Redis 存的是**跨实例共享**的数据，但网络抖动可能不可用
- 内存存的是**当前实例**的数据，永远可用
- `available` 属性永远返回 `True`，调用方无需判断

### 4.4 文本清洗的细节

```python
# text_cleaner.py — 三步关键处理

# 1. 中文行间换行合并（PDF 提取常见问题）
text = re.sub(r'(?<=[一-鿿])\n(?=[一-鿿\w\d])', '', text)
# "张\n三" → "张三"（中文字符间的换行去掉）

# 2. 合并连续空行为单个空行
if not line:
    if not prev_empty: result_lines.append('')
    
# 3. 压缩多余空格
text = re.sub(r' {2,}', ' ', text)
```

### 4.5 PDF 解析的三级校验

```python
# pdf_parser.py:28-35
def validate_pdf(file_content, filename):
    # Level 1: 文件名后缀
    if not filename.lower().endswith('.pdf'): raise PDFParseError(...)
    # Level 2: 文件大小
    if len(file_content) > MAX_FILE_SIZE: raise PDFParseError(...)
    # Level 3: 文件魔数（字节级验证）
    if not file_content.startswith(b'%PDF'): raise PDFParseError(...)
```

为什么不用 `python-magic` 或 MIME type？因为这三级校验已经覆盖了 99% 的恶意文件，且不引入额外依赖。

### 4.6 性能优化思路

| 环节 | 当前方案 | 优化方向 |
|------|---------|---------|
| PDF 解析 | PyMuPDF 逐页同步 | 大文件分页异步 |
| AI 调用 | 同步等待 | 对于批量简历，可并发调用 |
| 缓存命中 | 直接返回 | 已实现（跳过 AI 调用，省 90% 时间） |
| 冷启动 | FC 冷启动 ≤1s | 设置最小实例数=1，热启动 <100ms |

## 六、面试高频问答

### Q1: "介绍一下你的项目"

> **建议回答结构**（30秒版本）：
>
> 我用 FastAPI + 通义千问 + 阿里云 FC 做了一个 AI 简历分析系统。上传 PDF 简历后，AI 自动提取姓名、技能、项目经历等关键信息，然后根据岗位描述进行智能匹配评分。前端部署在 GitHub Pages，后端在阿里云函数计算。全程有缓存机制，相同简历不会重复调用 AI。

### Q2: "上传一张图片 PDF（扫描件），系统会怎样？"

> **考察点**：是否考虑过边界情况
>
> **代码位置**：`pdf_parser.py:68-69`
>
> PyMuPDF 无法从图片 PDF 提取文本 → `cleaned_text` 为空 → 抛出 `"PDF 文本清洗后为空，可能是扫描版 PDF 或图片简历"`，HTTP 400。
>
> **延伸**：如果要支持，可以加 OCR —— 用 `pdf2image` 转图片 → 通义千问 VL 多模态识别，或者用 PaddleOCR 做本地 OCR。

### Q3: "AI 返回格式错误怎么办？你的系统怎么处理的？"

> **考察点**：容错设计
>
> **代码位置**：`ai_extractor.py:92-114`
>
> 三层容错：
> 1. JSON 解析异常 → 重试（最多 2 次）
> 2. 重试全部失败 → 降级为正则规则提取（邮箱 `r'[\w.%+-]+@[\w.-]+\.\w{2,}'`、手机号 `r'1[3-9]\d{9}'`、姓名 `r'^[一-鿿]{2,4}$'`）
> 3. 规则也失败 → 返回空字段，不报错中断
>
> 另外，即使 AI 成功，也会用正则做**补充校验**（`ai_extractor.py:113-116`）——AI 漏掉的邮箱/电话用正则补齐。

### Q4: "匹配评分 83 分是怎么算出来的？"

> **考察点**：是否理解自己的评分逻辑
>
> **代码位置**：`matcher.py:147-149`
>
> ```python
> 综合分数 = skill_match × 0.5 + experience_relevance × 0.3 + education_match × 0.2
> ```
>
> AI 评分 × 60%（精准但慢）+ 规则评分 × 40%（快但粗糙）
>
> 规则评分逻辑 (`matcher.py:65-89`)：
> - 技能匹配：简历技能和 JD 关键词的交集率
> - 经验相关性：应届→30分，1-2年→50分，3-5年→75分，5年+→90分
> - 学历匹配：博士→95，硕士→85，本科→75，大专→50
>
> **为什么选这些权重？** 技能是最刚性的匹配条件，经验次之，学历是参考项。这些权重可以调整——面试时可以承认权重需要根据实际数据校准。

### Q5: "Python import 有什么坑？你项目中遇到过吗？"

> **考察点**：Python 底层理解
>
> **代码位置**：`app/routers/resume.py:10` vs commit `8a5dc80`
>
> 遇到过经典问题：`from app.services.cache import cache_service` 在 FastAPI 的 `lifespan` 场景下不生效。
>
> **原因**：`from-import` 是**值绑定**——import 时 `cache_service = None`，之后 lifespan 里改了模块属性 `cache_mod.cache_service = new_instance`，但 router 里的本地名还是 `None`。
>
> **解决**：改为 `import app.services.cache as cache_mod`，每次用 `cache_mod.cache_service` 动态取值。
>
> 这是 Python 一个不常被注意但很致命的特性。

### Q6: "如果 1000 人同时上传简历，你的系统扛得住吗？"

> **考察点**：对 Serverless 的理解 + 瓶颈意识
>
> - FC 弹性实例会自动扩容，每个实例处理 20 个并发请求
> - FastAPI 是异步的，I/O 等待时不阻塞线程
> - 真正的瓶颈是通义千问 API 的 QPS 限制（qwen-turbo 约 20 QPS）
> - 优化：缓存命中直接返回（不调 AI）→ 如果有热点文件（大量相同简历），缓存可扛掉 90% 流量
> - 进一步：异步消息队列削峰填谷

### Q7: "README 里写了用 PyMuPDF，为什么不选 pdfplumber？"

> **考察点**：技术选型的 trade-off 意识
>
> | | PyMuPDF | pdfplumber |
> |--|---------|------------|
> | 速度 | 快 (C 扩展) | 慢 (纯 Python) |
> | 中文 | 好 | 好 |
> | 依赖 | 需要 libmupdf | 无系统依赖 |
> | 功能 | 提取 + 渲染 | 仅提取 |
>
> 选 PyMuPDF 是因为速度 — Serverless 按执行时间计费，解析越快成本越低。

### Q8: "你的前端为什么不用 React/Vue？"

> **考察点**：简化取舍的能力
>
> 这个项目只有一个页面、两个表单、几个信息展示卡片。React/Vue 的组件化和状态管理优势在这里体现不出来，反而增加了构建步骤和部署复杂度。原生 HTML/CSS/JS 零依赖、零构建、直接部署，符合 "用最简单的工具解决问题" 的原则。

### Q9: "如果让你继续做，你会加什么功能？"

> **建议回答**：
> 1. OCR 支持（处理图片 PDF 和拍照简历）
> 2. 批量上传（一次传 50 份简历，并发处理）
> 3. 简历对比模式（两份简历同岗位对比排名）
